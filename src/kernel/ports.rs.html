1//! # x86 I/O Port Access
2//!
3//! This module provides low-level access to the x86/x86-64 I/O port address space,
4//! offering safe wrappers around the `in` and `out` assembly instructions for
5//! legacy device communication. It enables kernel drivers to interact with
6//! hardware that uses port-mapped I/O rather than memory-mapped I/O (MMIO).
7//!
8//! ## Overview
9//!
10//! The x86 architecture provides two distinct address spaces for device communication:
11//! - **I/O Port Space**: Legacy 16-bit address space (0x0000-0xFFFF) accessed via
12//!   special `in`/`out` instructions
13//! - **Memory Space**: Normal memory addresses accessed via standard load/store
14//!   instructions (MMIO)
15//!
16//! This module handles the I/O port space, which is primarily used by legacy
17//! devices and some system controllers that maintain backward compatibility.
18//!
19//! ## Architecture Details
20//!
21//! ### I/O Port Space Characteristics
22//! * **16-bit Addressing**: Port numbers range from 0x0000 to 0xFFFF
23//! * **Separate Address Space**: Completely distinct from physical memory
24//! * **Special Instructions**: Requires `in`/`out` instructions, not memory loads/stores
25//! * **Privilege Controlled**: Access controlled by CPL, IOPL, and I/O permission bitmap
26//!
27//! ### Common Port Ranges
28//! ```text
29//! 0x0000-0x001F   DMA Controllers
30//! 0x0020-0x0021   Programmable Interrupt Controller (PIC) #1
31//! 0x0040-0x0043   Programmable Interval Timer (PIT)
32//! 0x0060-0x0064   Keyboard Controller
33//! 0x0070-0x0071   CMOS/RTC
34//! 0x00A0-0x00A1   PIC #2
35//! 0x00F0-0x00FF   Math Coprocessor
36//! 0x0170-0x0177   Secondary IDE Controller
37//! 0x01F0-0x01F7   Primary IDE Controller
38//! 0x0278-0x027A   Parallel Port #2
39//! 0x02E8-0x02EF   Serial Port #4
40//! 0x02F8-0x02FF   Serial Port #2
41//! 0x0378-0x037A   Parallel Port #1
42//! 0x03E8-0x03EF   Serial Port #3
43//! 0x03F0-0x03F7   Floppy Disk Controller
44//! 0x03F8-0x03FF   Serial Port #1
45//! ```
46//!
47//! ## Available Operations
48//!
49//! ### Byte Operations
50//! * [`outb`] - Write a single byte to an I/O port
51//! * [`inb`] - Read a single byte from an I/O port
52//!
53//! ### Usage Example
54//! ```rust
55//! use crate::ports::{inb, outb};
56//!
57//! // Read keyboard status (port 0x64)
58//! let status = unsafe { inb(0x64) };
59//!
60//! // Write to serial port (port 0x3F8)
61//! unsafe { outb(0x3F8, b'H') };
62//! ```
63//!
64//! ## Safety Requirements
65//!
66//! All I/O port operations are inherently unsafe due to their direct hardware
67//! interaction and potential system impact. Callers must ensure:
68//!
69//! ### Privilege Requirements
70//! * **Ring 0 Execution**: Most secure when running in kernel mode (CPL 0)
71//! * **I/O Permission**: If not in ring 0, IOPL bits or I/O permission bitmap
72//!   must allow access to the specific port
73//!
74//! ### Hardware Safety
75//! * **Correct Port**: Target the intended device register, not arbitrary addresses
76//! * **Device Presence**: Ensure the device exists and is properly initialized
77//! * **Protocol Compliance**: Follow device-specific communication protocols
78//! * **Timing Requirements**: Respect device timing constraints and handshakes
79//!
80//! ### Concurrency Safety
81//! * **Mutual Exclusion**: Coordinate with interrupt handlers and other threads
82//! * **Atomic Sequences**: Protect multi-step device interactions from interruption
83//! * **Driver Coordination**: Ensure only one driver controls each device
84//!
85//! ### Memory Ordering
86//! * **I/O Ordering**: `in`/`out` instructions are ordered relative to each other
87//! * **Memory Barriers**: Insert appropriate fences when coordinating with MMIO
88//! * **Compiler Barriers**: Prevent unwanted optimization of I/O sequences
89//!
90//! ## Design Philosophy
91//!
92//! This module follows a minimal, explicit approach:
93//! * **Thin Wrappers**: Direct exposure of hardware capabilities without abstraction
94//! * **Safety Documentation**: Comprehensive safety requirements rather than runtime checks
95//! * **Performance Focus**: Zero-overhead inline assembly with optimal instruction selection
96//! * **Explicit Unsafe**: All hardware interaction requires explicit acknowledgment of risks
97//!
98//! ## Future Extensions
99//!
100//! Additional I/O operations may be added as needed:
101//! * 16-bit operations (`inw`/`outw`) for word-sized transfers
102//! * 32-bit operations (`inl`/`outl`) for double-word transfers
103//! * String operations (`insb`/`outsb`) for bulk transfers
104//! * I/O delay helpers for timing-sensitive devices

106/// Write one byte to an I/O port (x86).
107///
108/// Low-level helper for devices that live in the legacy **I/O port space**
109/// (not MMIO). Uses `out dx, al`.
110///
111/// # Safety
112/// You must uphold **all** of the following:
113/// - **Privilege:** Execute at CPL0 **or** have I/O permission (IOPL/IO bitmap)
114///   that allows access to `port`. Otherwise the CPU raises `#GP`.
115/// - **Correct port:** `port` must belong to the intended device and be in a
116///   valid state for this write. Writing the wrong port or wrong value can wedge
117///   the device or the system (e.g., disabling the PIC, reprogramming timers).
118/// - **Device presence:** The target device must exist and be decoded on the
119///   bus. Some platforms hang on accesses to nonexistent ports.
120/// - **Concurrency:** Coordinate with interrupt handlers and other CPUs/threads
121///   that touch the same device/port. Use your driver’s locking/serialization
122///   so register-level protocols aren’t violated.
123/// - **Ordering:** `out` orders with respect to other I/O instructions to the
124///   same device but is **not** a general memory fence. If you need ordering
125///   with normal memory (e.g., MMIO buffers or shared memory), add an
126///   appropriate compiler/CPU fence around calls.
127/// - **Environment:** Only use on `x86/x86_64` with an I/O port bus. Never use
128///   for **memory-mapped** devices (MMIO).
129#[inline]
130pub unsafe fn outb(port: u16, val: u8) {
131    unsafe {
132        core::arch::asm!("out dx, al", in("dx") port, in("al") val, options(nomem, nostack, preserves_flags));
133    }
134}

136/// Read one byte from an I/O port (x86).
137///
138/// Low-level helper for devices that live in the legacy **I/O port space**.
139/// Uses `in al, dx`.
140///
141/// # Safety
142/// You must uphold **all** of the following:
143/// - **Privilege:** Execute at CPL0 **or** have I/O permission (IOPL/IO bitmap)
144///   that allows access to `port`; otherwise the CPU raises `#GP`.
145/// - **Correct port:** `port` must be a readable register of the intended
146///   device; reading from the wrong port can yield undefined garbage or stall
147///   the device’s protocol.
148/// - **Device presence:** The target device must exist and be decoding the
149///   address. Accesses to nonexistent ports may fault or hang on some systems.
150/// - **Concurrency:** Coordinate with interrupt handlers/other CPUs that
151///   manipulate the same device/port to avoid tearing multi-step handshakes.
152/// - **Ordering:** `in` orders with other I/O instructions but is **not** a
153///   general memory fence. If you must order this read with normal memory
154///   operations (e.g., reading a status port then consuming an MMIO buffer),
155///   insert the appropriate compiler/CPU fence.
156/// - **Environment:** Only for `x86/x86_64` I/O port space. Do not use for MMIO.
157#[inline]
158pub unsafe fn inb(port: u16) -> u8 {
159    let mut v: u8;
160    unsafe {
161        core::arch::asm!("in al, dx", in("dx") port, out("al") v, options(nomem, nostack, preserves_flags));
162    }
163    v
164}
