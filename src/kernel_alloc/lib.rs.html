1//! # Kernel Memory Allocation and Virtual Memory Management
2//!
3//! This crate provides the core memory allocation infrastructure for the kernel,
4//! implementing both physical frame allocation and virtual memory management
5//! capabilities. It serves as the foundation for all memory operations in the
6//! operating system, from initial bootstrap through runtime operation.
7//!
8//! ## Architecture Overview
9//!
10//! The memory management system is built on a three-layer architecture:
11//!
12//! ```text
13//! ┌─────────────────────────────────────────────────────┐
14//! │                Virtual Memory Manager (VMM)         │
15//! │    • Page table manipulation                        │
16//! │    • Virtual address space management               │
17//! │    • User/kernel space separation                   │
18//! └─────────────────┬───────────────────────────────────┘
19//!                   │
20//! ┌─────────────────▼───────────────────────────────────┐
21//! │              Physical Mapper                        │
22//! │    • Physical-to-virtual address translation        │
23//! │    • HHDM (Higher Half Direct Mapping)              │
24//! │    • Safe pointer conversion                        │
25//! └─────────────────┬───────────────────────────────────┘
26//!                   │
27//! ┌─────────────────▼───────────────────────────────────┐
28//! │           Physical Frame Allocator                  │
29//! │    • 4KiB page frame management                     │
30//! │    • Bitmap-based free/used tracking                │
31//! │    • No-heap allocation strategy                    │
32//! └─────────────────────────────────────────────────────┘
33//! ```
34//!
35//! ## Core Components
36//!
37//! ### Physical Frame Allocator ([`frame_alloc`])
38//!
39//! Manages the allocation and deallocation of 4KiB physical memory frames:
40//! * **Bitmap Management**: Efficient tracking of free/used frames using bit arrays
41//! * **No-Heap Design**: Self-contained implementation requiring no dynamic allocation
42//! * **Fixed Region**: Manages a predefined region of physical memory (currently 512 MiB)
43//! * **Early Boot Support**: Suitable for use before full memory management is available
44//!
45//! Key features:
46//! - O(1) allocation when frames are available
47//! - Simple bitmap-based tracking for reliability
48//! - Configurable memory region boundaries
49//! - Integration with kernel memory layout
50//!
51//! ### Physical Mapper ([`phys_mapper`])
52//!
53//! Provides safe conversion between physical addresses and virtual pointers:
54//! * **HHDM Support**: Higher Half Direct Mapping for efficient address translation
55//! * **Safe Abstractions**: Type-safe pointer conversions with lifetime management
56//! * **Page Table Access**: Enables manipulation of physical page table structures
57//! * **Cross-Platform**: Abstracts physical memory access patterns
58//!
59//! Key capabilities:
60//! - Physical address to virtual pointer conversion
61//! - Support for different mapping strategies
62//! - Safe dereferencing of physical memory
63//! - Integration with page table manipulation
64//!
65//! ### Virtual Memory Manager ([`vmm`])
66//!
67//! Coordinates virtual address space management and page table operations:
68//! * **Address Space Management**: Separate user and kernel virtual address spaces
69//! * **Page Table Manipulation**: Creation, modification, and destruction of mappings
70//! * **Memory Protection**: Configurable page permissions (read, write, execute, user)
71//! * **Anonymous Mapping**: On-demand allocation of virtual memory regions
72//! * **Region Management**: Bulk operations for mapping and unmapping areas
73//!
74//! Advanced features:
75//! - Guard page support for stack overflow detection
76//! - Bulk mapping operations for performance
77//! - TLB management and invalidation
78//! - User/kernel space isolation
79//!
80//! ## Memory Layout Integration
81//!
82//! The crate integrates with the kernel's memory layout defined in `kernel-info`:
83//!
84//! ```text
85//! Virtual Address Space Layout:
86//! 0x0000_0000_0000_0000 ┌─────────────────────────────────┐
87//!                       │        User Space               │
88//!                       │  (Applications, libraries)      │
89//! LAST_USERSPACE_ADDRESS├─────────────────────────────────┤
90//!                       │        Guard Region             │
91//! HHDM_BASE             ├─────────────────────────────────┤
92//!                       │   Higher Half Direct Map        │
93//!                       │  (Physical memory access)       │
94//! KERNEL_BASE           ├─────────────────────────────────┤
95//!                       │       Kernel Space              │
96//!                       │  (Kernel code and data)         │
97//! 0xFFFF_FFFF_FFFF_FFFF └─────────────────────────────────┘
98//! ```
99//!
100//! ## Safety Model
101//!
102//! The memory management system employs multiple layers of safety:
103//!
104//! ### Type Safety
105//! * **Address Types**: Distinct types for physical and virtual addresses
106//! * **Page Alignment**: Compile-time guarantees for page-aligned operations
107//! * **Lifetime Management**: Rust ownership prevents use-after-free errors
108//!
109//! ### Runtime Safety
110//! * **Bounds Checking**: Validation of memory region boundaries
111//! * **Permission Enforcement**: Hardware-backed memory protection
112//! * **Guard Pages**: Overflow detection through unmapped regions
113//! * **TLB Synchronization**: Proper cache invalidation on mapping changes
114//!
115//! ### Concurrency Safety
116//! * **Atomic Operations**: Thread-safe allocation algorithms
117//! * **Critical Sections**: Protection of shared data structures
118//! * **Lock-Free Paths**: Performance optimization for common operations
119//!
120//! ## Usage Patterns
121//!
122//! ### Basic Physical Allocation
123//! ```rust
124//! use kernel_alloc::frame_alloc::BitmapFrameAlloc;
125//! use kernel_vmem::PhysFrameAlloc;
126//!
127//! let mut allocator = BitmapFrameAlloc::new();
128//! if let Some(frame) = allocator.alloc_4k() {
129//!     // Use the physical frame
130//!     allocator.free_4k(frame);
131//! }
132//! ```
133//!
134//! ### Virtual Memory Management
135//! ```rust,no_run
136//! use kernel_alloc::{phys_mapper::HhdmPhysMapper, vmm::Vmm};
137//! use kernel_alloc::frame_alloc::BitmapFrameAlloc;
138//!
139//! let mapper = HhdmPhysMapper;
140//! let mut allocator = BitmapFrameAlloc::new();
141//! let mut vmm = unsafe { Vmm::from_current(&mapper, &mut allocator) };
142//!
143//! // Map virtual memory regions, manage page tables, etc.
144//! ```
145//!
146//! ## Performance Characteristics
147//!
148//! * **Physical Allocation**: O(n) worst case, O(1) typical case
149//! * **Virtual Mapping**: O(1) for single pages, O(n) for regions
150//! * **Address Translation**: O(1) with HHDM
151//! * **Memory Overhead**: ~1 bit per 4KiB frame for allocation tracking
152//!
153//! ## Integration Points
154//!
155//! This crate integrates with several other kernel components:
156//! * **kernel-vmem**: Core virtual memory abstractions and types
157//! * **kernel-info**: Memory layout constants and configuration
158//! * **kernel-sync**: Synchronization primitives for thread safety
159//!
160//! The modular design enables testing, portability, and future enhancements
161//! while maintaining clear separation of concerns between different memory
162//! management responsibilities.
