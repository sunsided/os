1//! # Kernel Configuration and Boot Interface
2//!
3//! This crate defines the core data structures, memory layout constants, and
4//! boot interface contracts that govern the kernel's initialization and runtime
5//! operation. It serves as the authoritative source for system-wide configuration
6//! and provides stable ABI definitions for communication between the bootloader
7//! and kernel.
8//!
9//! ## Overview
10//!
11//! The kernel requires precise coordination between multiple components during
12//! boot and runtime operation. This crate centralizes critical configuration
13//! information that must be shared across the bootloader, kernel, and various
14//! kernel subsystems, ensuring consistency and preventing configuration drift.
15//!
16//! ## Architecture
17//!
18//! The crate is organized into two primary modules:
19//!
20//! ### Boot Information ([`boot`])
21//! Defines the bootloader-to-kernel handoff interface:
22//! * **Kernel Entry Point**: Function signature and calling convention
23//! * **Boot Data Structures**: Memory map, ACPI information, framebuffer details
24//! * **ABI Stability**: C-compatible structures for cross-component communication
25//! * **UEFI Integration**: Direct compatibility with UEFI GOP and memory services
26//!
27//! ### Memory Layout ([`memory`])
28//! Establishes the kernel's virtual memory architecture:
29//! * **Address Space Layout**: User/kernel space boundaries and reserved regions
30//! * **Higher Half Design**: Kernel execution at high virtual addresses
31//! * **Physical Memory Mapping**: HHDM (Higher Half Direct Mapping) configuration
32//! * **Stack and Code Placement**: Compile-time memory organization
33//!
34//! ## Virtual Memory Architecture
35//!
36//! The kernel employs a higher-half design with carefully planned address space layout:
37//!
38//! ```text
39//! Virtual Address Space Layout (64-bit):
40//!
41//! 0x0000_0000_0000_0000 ┌─────────────────────────────────┐
42//!                       │                                 │
43//!                       │         User Space              │
44//!                       │    (Applications & Libraries)   │
45//!                       │                                 │
46//! LAST_USERSPACE_ADDRESS├─────────────────────────────────┤ 0xffff_efff_ffff_ffff
47//!                       │        Guard Region             │
48//! HHDM_BASE             ├─────────────────────────────────┤ 0xffff_8880_0000_0000
49//!                       │   Higher Half Direct Mapping    │
50//!                       │   (Physical Memory Access)      │
51//! KERNEL_BASE           ├─────────────────────────────────┤ 0xffff_ffff_8000_0000
52//!                       │       Kernel Text & Data        │
53//!                       │     (Kernel Executable)         │
54//! 0xFFFF_FFFF_FFFF_FFFF └─────────────────────────────────┘
55//! ```
56//!
57//! ### Design Principles
58//! * **Canonical Addressing**: All kernel addresses use the canonical higher half
59//! * **Guard Regions**: Large unmapped areas prevent accidental user/kernel overlap
60//! * **Direct Mapping**: HHDM enables efficient physical memory access
61//! * **Fixed Layout**: Compile-time constants enable static optimization
62//!
63//! ## Boot Protocol
64//!
65//! The bootloader-to-kernel handoff follows a well-defined protocol:
66//!
67//! ### Entry Point Convention
68//! ```rust
69//! # use kernel_info::boot::KernelBootInfo;
70//! pub type KernelEntryFn = extern "win64" fn(*const KernelBootInfo) -> !;
71//! ```
72//!
73//! * **Calling Convention**: Windows x64 ABI for UEFI compatibility
74//! * **Parameter**: Single pointer to boot information structure
75//! * **No Return**: Kernel assumes control permanently
76//!
77//! ### Boot Information Structure
78//! The [`KernelBootInfo`](boot::KernelBootInfo) structure provides:
79//! * **Memory Map**: UEFI memory map for physical memory management
80//! * **ACPI Root**: RSDP address for hardware discovery
81//! * **Framebuffer**: GOP framebuffer for early graphics output
82//!
83//! ## Physical Memory Layout
84//!
85//! The kernel's physical memory placement is coordinated with the linker:
86//!
87//! ```text
88//! Physical Memory Layout:
89//! 0x0000_0000 ┌─────────────────────────────────┐
90//!             │     Low Memory (< 1MiB)         │
91//!             │  (BIOS, VGA, DMA buffers)       │
92//! PHYS_LOAD   ├─────────────────────────────────┤ 0x0010_0000 (1 MiB)
93//!             │       Kernel Image              │
94//!             │   (Text, Data, BSS)             │
95//!             ├─────────────────────────────────┤
96//!             │    Available RAM                │
97//!             │  (Managed by allocator)         │
98//!             └─────────────────────────────────┘
99//! ```
100//!
101//! * **Low Memory Avoidance**: Kernel loads at 1 MiB to avoid legacy conflicts
102//! * **Identity Mapping**: Small identity region for paging setup
103//! * **Linker Integration**: Constants used in kernel linker script
104//!
105//! ## Configuration Management
106//!
107//! ### Compile-Time Constants
108//! All layout constants are `const` values computed at compile time:
109//! * **Memory Safety**: Compile-time assertions prevent invalid configurations
110//! * **Performance**: No runtime computation of layout information
111//! * **Consistency**: Single source of truth for all kernel components
112//!
113//! ### Build Integration
114//! The constants are consumed by the kernel's `build.rs`:
115//! * **Linker Script Generation**: Dynamic linker script creation
116//! * **Symbol Definition**: Automatic symbol generation for assembly
117//! * **Validation**: Build-time verification of layout constraints
118//!
119//! ## ABI Compatibility
120//!
121//! All public structures maintain strict ABI compatibility:
122//!
123//! ### C Representation
124//! * **`#[repr(C)]`**: Predictable memory layout for cross-language use
125//! * **Fixed-Size Types**: Explicit integer sizes for platform independence
126//! * **No Rust Enums**: Simple discriminated unions for ABI stability
127//!
128//! ### UEFI Integration
129//! * **GOP Compatibility**: Direct mapping from UEFI graphics structures
130//! * **Memory Map Format**: Native UEFI memory descriptor compatibility
131//! * **Calling Conventions**: UEFI-compatible function signatures
132//!
133//! ## Safety Guarantees
134//!
135//! This crate maintains several safety invariants:
136//!
137//! ### Memory Layout Safety
138//! * **No Overlaps**: Compile-time verification of non-overlapping regions
139//! * **Canonical Addresses**: All kernel addresses are in canonical form
140//! * **Alignment Requirements**: Page and stack alignment guarantees
141//!
142//! ### ABI Safety
143//! * **Stable Layouts**: No unexpected structure reorganization
144//! * **Version Compatibility**: Backward-compatible structure evolution
145//! * **No Unsafe Code**: Marked `#![deny(unsafe_code)]` for safety assurance
146//!
147//! ## Usage Patterns
148//!
149//! ### Build Script Integration
150//! ```rust
151//! // In build.rs
152//! use kernel_info::memory::{KERNEL_BASE, PHYS_LOAD};
153//!
154//! println!("cargo:rustc-link-arg=--defsym=KERNEL_BASE={:#x}", KERNEL_BASE.as_u64());
155//! println!("cargo:rustc-link-arg=--defsym=PHYS_LOAD={:#x}", PHYS_LOAD.as_u64());
156//! ```
157//!
158//! ### Bootloader Integration
159//! ```rust,ignore
160//! use kernel_info::boot::{KernelBootInfo, KernelEntryFn};
161//!
162//! let boot_info = KernelBootInfo {
163//!     mmap: /* memory map info */,
164//!     rsdp_addr: /* ACPI root */,
165//!     fb: /* framebuffer info */,
166//! };
167//!
168//! let kernel_entry: KernelEntryFn = /* kernel entry point */;
169//! kernel_entry(&boot_info); // Transfer control to kernel
170//! ```
171//!
172//! This crate serves as the foundation for all kernel configuration, ensuring
173//! consistency across the entire operating system implementation while maintaining
174//! clear interfaces between system components.
