1//! # UEFI Bootloader for Higher-Half x86-64 Kernel
2//!
3//! This UEFI application serves as the bootloader for the operating system kernel,
4//! handling the complex transition from UEFI firmware environment to a fully
5//! operational higher-half x86-64 kernel. It orchestrates ELF loading, memory
6//! management setup, and the critical handoff to kernel execution.
7//!
8//! ## Overview
9//!
10//! Modern operating systems require sophisticated bootloaders that can:
11//! - Load and parse complex executable formats
12//! - Configure virtual memory management
13//! - Gather system information from firmware
14//! - Safely transition between execution environments
15//!
16//! This UEFI loader implements all these capabilities while maintaining compatibility
17//! with the UEFI specification and providing a robust foundation for kernel startup.
18//!
19//! ## Boot Process Architecture
20//!
21//! The bootloader follows a carefully orchestrated sequence to transition from
22//! UEFI firmware control to autonomous kernel operation:
23//!
24//! ```text
25//! UEFI Firmware Boot
26//!         ↓
27//! ┌─────────────────────────────────────────────┐
28//! │              UEFI Loader                    │
29//! ├─────────────────────────────────────────────┤
30//! │  1. Environment Setup                       │
31//! │     • Initialize logging and allocator      │
32//! │     • Configure UEFI boot services          │
33//! │  2. Kernel Loading                          │
34//! │     • Parse kernel.elf file                 │
35//! │     • Load PT_LOAD segments to memory       │
36//! │     • Resolve higher-half addresses         │
37//! │  3. System Discovery                        │
38//! │     • Obtain framebuffer configuration      │
39//! │     • Locate ACPI RSDP                      │
40//! │     • Gather memory map information         │
41//! │  4. Virtual Memory Setup                    │
42//! │     • Create kernel page tables             │
43//! │     • Map kernel segments at higher-half    │
44//! │     • Establish HHDM mapping                │
45//! │     • Set up trampoline stack               │
46//! │  5. Environment Transition                  │
47//! │     • Exit UEFI boot services               │
48//! │     • Enable memory protection features     │
49//! │     • Switch to kernel page tables          │
50//! │  6. Kernel Handoff                          │
51//! │     • Jump to kernel entry point            │
52//! │     • Transfer boot information             │
53//! └─────────────────────────────────────────────┘
54//!         ↓
55//! Kernel Execution (Higher-Half)
56//! ```
57//!
58//! ## Key Components
59//!
60//! ### ELF Loading and Processing
61//! * **File System Access**: Load `kernel.elf` from UEFI ESP filesystem
62//! * **ELF64 Parsing**: Parse program headers and extract loadable segments
63//! * **Higher-Half Mapping**: Resolve virtual addresses to higher-half locations
64//! * **Physical Placement**: Load segments at appropriate physical addresses
65//! * **Entry Point Resolution**: Extract kernel entry point for execution transfer
66//!
67//! ### Memory Management Setup
68//! * **Page Table Creation**: Build complete x86-64 page table hierarchy
69//! * **Kernel Segment Mapping**: Map all kernel segments with correct permissions
70//! * **HHDM Establishment**: Create Higher Half Direct Mapping for physical access
71//! * **Identity Mapping**: Maintain low-memory identity map for transition code
72//! * **Protection Features**: Enable NX-bit, write protection, and global pages
73//!
74//! ### System Information Gathering
75//! * **UEFI Memory Map**: Capture complete physical memory layout
76//! * **Graphics Configuration**: Obtain GOP framebuffer details
77//! * **ACPI Discovery**: Locate RSDP for hardware enumeration
78//! * **Boot Information**: Package data for kernel consumption
79//!
80//! ### Transition Management
81//! * **Boot Services Exit**: Safely terminate UEFI boot services
82//! * **Trampoline Execution**: Execute page table switch code
83//! * **Stack Setup**: Provide kernel with initial execution stack
84//! * **Control Transfer**: Jump to kernel entry point with proper ABI
85//!
86//! ## Virtual Memory Layout
87//!
88//! The loader establishes a sophisticated virtual memory layout for the kernel:
89//!
90//! ```text
91//! Virtual Address Space (Post-Transition):
92//!
93//! 0x0000_0000_0000_0000 ┌────────────────────────────────┐
94//!                       │     Identity Mapped Region     │
95//!                       │   (UEFI transition code)       │ 2 MiB
96//! 0x0000_0000_0020_0000 ├────────────────────────────────┤
97//!                       │        Unmapped Space          │
98//!                       │                                │
99//! HHDM_BASE             ├────────────────────────────────┤ 0xffff_8880_0000_0000
100//!                       │   Higher Half Direct Mapping   │
101//!                       │    (All Physical Memory)       │
102//! KERNEL_BASE           ├────────────────────────────────┤ 0xffff_ffff_8000_0000
103//!                       │       Kernel Text (.text)      │
104//!                       │       Kernel Data (.data)      │
105//!                       │       Kernel BSS  (.bss)       │
106//!                       │       Kernel Stacks            │
107//! 0xFFFF_FFFF_FFFF_FFFF └────────────────────────────────┘
108//! ```
109//!
110//! ### Memory Protection Attributes
111//! * **Kernel Code**: Read + Execute, Global, Supervisor-only
112//! * **Kernel Data**: Read + Write + NX, Global, Supervisor-only
113//! * **HHDM Region**: Read + Write + NX, Global, Supervisor-only
114//! * **Identity Map**: Read + Execute, Global (temporary)
115//!
116//! ## ELF Loading Strategy
117//!
118//! The loader implements sophisticated ELF64 handling:
119//!
120//! ### Load Address Calculation
121//! ```text
122//! ELF Virtual Address (VMA) → Physical Load Address (LMA)
123//!
124//! For higher-half kernel:
125//! LMA = VMA - KERNEL_BASE
126//! Physical Address = PHYS_LOAD + LMA
127//!
128//! Example:
129//! VMA = 0xffff_ffff_8010_0000 (kernel text)
130//! LMA = 0x0010_0000
131//! Physical = 0x0010_0000 + 0x0010_0000 = 0x0010_0000
132//! ```
133//!
134//! ### Segment Processing
135//! 1. **Parse `PT_LOAD` Headers**: Extract loadable segment information
136//! 2. **Calculate Addresses**: Resolve virtual-to-physical mappings
137//! 3. **Allocate Memory**: Reserve physical pages for segment content
138//! 4. **Copy Content**: Transfer ELF data to allocated memory
139//! 5. **Zero BSS**: Initialize uninitialized data sections
140//!
141//! ## UEFI Integration
142//!
143//! ### Boot Services Utilization
144//! * **Memory Allocation**: Use UEFI allocator for temporary structures
145//! * **File Access**: Read kernel image from ESP filesystem
146//! * **Graphics Setup**: Configure GOP framebuffer for kernel graphics
147//! * **Protocol Access**: Interact with UEFI configuration tables
148//!
149//! ### Exit Strategy
150//! ```rust
151//! // Critical sequence for UEFI exit
152//! 1. Package all required information
153//! 2. Call ExitBootServices()
154//! 3. UEFI services now unavailable
155//! 4. Switch to kernel page tables
156//! 5. Jump to kernel entry point
157//! ```
158//!
159//! ## Trampoline Execution
160//!
161//! The transition to kernel requires careful stack and page table management:
162//!
163//! ### Assembly Transition Code
164//! ```asm
165//! cli                    ; Disable interrupts
166//! mov cr3, new_pml4      ; Switch page tables
167//! mov rsp, new_stack     ; Set kernel stack
168//! jmp kernel_entry       ; Transfer control
169//! ```
170//!
171//! ### Stack Management
172//! * **Trampoline Stack**: Temporary stack for page table switch
173//! * **Guard Pages**: Protect against stack overflow
174//! * **Alignment**: Maintain 16-byte stack alignment for ABI
175//! * **Identity Mapping**: Ensure trampoline code remains executable
176//!
177//! ## Error Handling
178//!
179//! The loader implements comprehensive error handling:
180//! * **ELF Validation**: Verify file format and architecture compatibility
181//! * **Memory Allocation**: Handle allocation failures gracefully
182//! * **Resource Cleanup**: Ensure proper cleanup on failure paths
183//! * **Status Reporting**: Provide meaningful error codes to firmware
184//!
185//! ## Security Considerations
186//!
187//! ### Memory Protection
188//! * **NX-bit Enforcement**: Prevent execution of data pages
189//! * **Write Protection**: Enable supervisor write protection
190//! * **Privilege Separation**: Establish kernel/user privilege levels
191//!
192//! ### Input Validation
193//! * **ELF Parsing**: Validate all ELF structures before use
194//! * **Address Bounds**: Verify all memory addresses are valid
195//! * **Size Limits**: Prevent integer overflow in size calculations
196//!
197//! ## Development and Debugging
198//!
199//! The loader integrates with QEMU debugging facilities:
200//! * **Debug Logging**: Comprehensive trace output for development
201//! * **Memory Layout Tracing**: Detailed memory allocation information
202//! * **Boot Information**: Complete system state at kernel handoff
203//! * **Error Diagnostics**: Detailed error reporting for troubleshooting
204//!
205//! ## Standards Compliance
206//!
207//! * **UEFI Specification**: Full compliance with UEFI 2.x requirements
208//! * **ELF64 Standard**: Correct interpretation of ELF file format
209//! * **x86-64 ABI**: Proper calling convention and stack management
210//! * **PE/COFF Format**: UEFI application binary format compliance

249#[entry]
250#[allow(clippy::too_many_lines)]
251fn efi_main() -> Status {
252    // Initialize logging and allocator helpers
253    if uefi::helpers::init().is_err() {
254        return Status::UNSUPPORTED;
255    }
256
257    let logger = UefiLogger::new(LevelFilter::Debug);
258    let logger = logger.init().expect("logger init");
259
260    info!("UEFI Loader reporting to QEMU");
261    info!("Attempting to load kernel.elf ...");
262
263    let elf_bytes = match load_file(cstr16!("\\EFI\\Boot\\kernel.elf")) {
264        Ok(bytes) => bytes,
265        Err(status) => {
266            info!("Failed to load kernel.elf. Exiting.");
267            return status;
268        }
269    };
270
271    // Parse ELF64, collect PT_LOAD segments and entry address
272    let Ok(parsed) = ElfHeader::parse_elf64(&elf_bytes) else {
273        info!("kernel.elf is not a valid x86_64 ELF64");
274        return Status::UNSUPPORTED;
275    };
276
277    info!("Loading kernel segments into memory ...");
278    let kernel_segments = match elf::loader::load_pt_load_segments_hi(&elf_bytes, &parsed) {
279        Ok(segments) => segments,
280        Err(e) => {
281            info!("Failed to load PT_LOAD segments: {e:?}");
282            return e.into();
283        }
284    };
285
286    info!(
287        "kernel.elf loaded successfully: entry={}, segments={}",
288        parsed.entry,
289        parsed.segments.len()
290    );
291
292    let fb = match get_framebuffer() {
293        Ok(fb) => fb,
294        Err(status) => {
295            return status;
296        }
297    };
298
299    // Locate RSDP before exiting boot services; if not found, set 0.
300    let rsdp_addr: u64 = find_rsdp_addr();
301
302    let boot_info = KernelBootInfo {
303        // Memory map fields are filled right after exit_boot_services returns the owned map:
304        mmap: MemoryMapInfo {
305            mmap_ptr: 0,
306            mmap_len: 0,
307            mmap_desc_size: 0,
308            mmap_desc_version: 0,
309        },
310        rsdp_addr,
311        fb,
312    };
313
314    // Heap-allocate and leak the boot info.
315    let boot_info = Box::leak(Box::new(boot_info));
316    info!("Kernel boot info: {:#?}", core::ptr::from_ref(boot_info));
317
318    // The trampoline code must also be mapped, otherwise we won't be able to execute it
319    // when switching the CR3 page tables.
320    let tramp_code_va = VirtualAddress::new(switch_to_kernel as usize as u64);
321    let tramp_code_len: usize = PAGE_SIZE; // should be enough
322
323    // Allocate a trampoline stack (with guard page)
324    debug!(
325        "Allocating trampoline stack for Kernel ({tramp_code_va}, {TRAMPOLINE_STACK_SIZE_BYTES} bytes)"
326    );
327    let (tramp_stack_base_phys, tramp_stack_top_va) =
328        alloc_trampoline_stack(TRAMPOLINE_STACK_SIZE_BYTES, true);
329
330    // Pass identity-mapped low pointer
331    let bi_ptr_va = VirtualAddress::from_ptr(core::ptr::from_ref::<KernelBootInfo>(boot_info));
332
333    // Build page tables
334    info!("Creating initial kernel page tables ...");
335    let Ok(pml4_phys) = create_kernel_pagetables(
336        &kernel_segments,
337        tramp_code_va,
338        tramp_code_len,
339        tramp_stack_base_phys,
340        TRAMPOLINE_STACK_SIZE_BYTES,
341        bi_ptr_va,
342    ) else {
343        uefi::println!("Failed to create kernel page tables");
344        return Status::OUT_OF_RESOURCES;
345    };
346
347    logger.exit_boot_services();
348    boot_info.mmap = match exit_boot_services() {
349        Ok(value) => value,
350        Err(value) => return value,
351    };
352
353    // Off we pop.
354    unsafe {
355        trace_boot_info(boot_info, bi_ptr_va, parsed.entry, tramp_stack_top_va);
356        enable_wp_nxe_pge();
357
358        // Activate our CR3 and jump to kernel entry (higher-half VA)
359        switch_to_kernel(
360            pml4_phys,
361            parsed.entry, // this is the higher-half VMA from ELF header
362            bi_ptr_va,
363            tramp_stack_top_va,
364        )
365    }
366}

368#[allow(clippy::items_after_statements)]
369unsafe fn enable_wp_nxe_pge() {
370    info!("Enabling supervisor write protection ...");
371    unsafe {
372        // CPU is already in long mode due to UEFI specs, i.e. paging bit is set, etc.
373        Cr0::load_unsafe()
374            .with_wp_write_protect(true)
375            .store_unsafe();
376    }
377
378    info!("Setting no-execute support for page tables (EFER.NXE) ...");
379    unsafe {
380        Efer::load_unsafe().with_nxe(true).store_unsafe();
381    }
382
383    info!("Enabling global pages ...");
384    unsafe {
385        Cr4::load_unsafe().with_pge(true).store_unsafe();
386    }
387}

389type PageTablePhysicalAddress = PhysicalAddress;
390type KernelVirtualAddress = VirtualAddress;
391type BootInfoVirtualAddress = VirtualAddress;
392type TrampolineStackVirtualAddress = VirtualAddress;

394/// Enter the kernel via a tiny trampoline.
395/// - `new_cr3`: phys addr of PML4 (4KiB aligned)
396/// - `kernel_entry`: higher-half VA (your `extern "win64" fn(*const BootInfo) -> !`)
397/// - `boot_info`: higher-half VA (or low VA if you pass identity-mapped pointer)
398/// - `tramp_stack_top`: VA of the top of the trampoline stack (identity-mapped in both maps)
399#[inline(never)]
400unsafe fn switch_to_kernel(
401    pml4_phys: PageTablePhysicalAddress,
402    kernel_entry_va: KernelVirtualAddress,
403    boot_info_ptr_va: BootInfoVirtualAddress,
404    tramp_stack_top_va: TrampolineStackVirtualAddress,
405) -> ! {
406    info!("UEFI is about to jump into Kernel land. Ciao Kakao ...");
407    unsafe {
408        core::arch::asm!(
409            "cli",
410            // Set up stack pointer
411            "mov    rsp, rdx",
412            // Set up page tables
413            "mov    cr3, rdi",
414            // Set up arguments for sysv64: rdi = boot_info, rsi = (unused), rdx = (unused)
415            "mov    rdi, r8",
416            // Set up kernel entry address in rax
417            "mov    rax, rsi",
418            // Align RSP down to 16-byte boundary
419            "and    rsp, -16",
420            // Emulate a CALL by pushing a dummy return address (kernel entry never returns)
421            "push   0",
422            "jmp    rax",
423            in("rdi") pml4_phys.as_u64(),
424            in("rsi") kernel_entry_va.as_u64(),
425            in("rdx") tramp_stack_top_va.as_u64(),
426            in("r8")  boot_info_ptr_va.as_u64(),
427            options(noreturn)
428        )
429    }
